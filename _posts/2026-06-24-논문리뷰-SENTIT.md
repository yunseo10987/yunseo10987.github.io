## 🦥 본문

## Intro.

**Background & Motivation**

VR, AR, 360도 비디오와 같은 차세대 대화형 애플리케이션들은 초저지연 인프라가 필요로 하지만 Bufferbloat 문제가 존재

- bufferbloat : 패킷 유실 방지와 처리량 극대화를 위해 버퍼를 과도하게 크게 설정한 결과, 큐잉 지연이 발생하는 현상
    - 패킷을 드랍하여 평균 패킷 지연 시간을 임계값 이하로 유지

**Limitations of the conventional approach**

1. ping, traceroute
    - ICMP 기반으로 네트워크 계층(L3)에서만 동작
    - 실제 데이터가 처리되는 전송 계층(L4) 이상의 지연은 측정 불가
2. 스택 내부 분석의 어려움 
    - 지연 시간을 정확히 파악하려면 프로토콜 스택 각 계층의 버퍼 체류 시간을 모두 확인해야 함
    - 루트 권한 및 네트워크 스택 전문 지식 필요
3. 프로토콜 전환
    - Bufferbloat는 주로 TCP에서 발생하기 때문에 UDP + 애플리케이션 계층 재설계로 우회
    - TCP의 신뢰성 있는 전송, 공정한 대역폭 공유 등의 장점을 포기
4. NIC 계층은 문제 없음
    - NIC의 IP 계층, 물리 계층은 TSO/GSO 기술로 지연이 거의 없고 그 위의 전송·애플리케이션 계층에 집중됨

**IDEA of ELEMENT**

리눅스의 user-space에서 접근 가능한 `tcp_info` 소켓 정보를 활용. 

- `tcp_info` : 확인된 바이트 수, 확인되지 않은 바이트 수, 타임스탬프 등 기본적인 TCP 통계
- 애플리케이션 계층 정보 : 송신 요청 시점, 전송량 등
- 두 정보를 결합하여 TCP 스택에서 발생하는 지연 시간 추론

**Contribution**

- 지연 시간 분해 : 전체 지연을 엔드호스트 지연 + 네트워크 지연으로 분리하여 원인 파악
- 사용자 수준의 라이브러리 제공 : 루트 권한 없이 적용 가능한 라이브러리 제공
    - 패킷 페이싱으로 데이터를 균일한 간격으로 전송하여 bufferbloat 억제
    - 동적 바인딩을 통해 transparent하게 적용하여 기존 애플리케이션 수정 없이 적용 가능
- 실증 구현 : ELEMENT를 적용한 실제 애플리케이션 구현 및 검증

## Latency in Real Network

지연의 원인은 Endhost에 있음을 보여주고, 최신의 기술로도 그 문제를 해결할 수 없음. 네트워크 지연과 엔드호스트 시스템 지연을 구분하지 못해 지연 시간을 성공적으로 줄이지 못하고 있음을 보여줌. 

**End-to-End Delay 측정 실험**

- Testbed : 송신자, 수신자, 그리고 그 사이에서 대역폭, 지연, 패킷 손실 등 네트워크 특성을 모방하는 WAN 에뮬레이터로 구성
    - WAN 에뮬레이터 : 인터넷 환경을 발생시키는 통신 상태를 구현하는 도구
- 세부 설정 : 리눅스 기본 버퍼 튜닝과 큐잉 방식(pfifo_fast)을 사용. 대역폭 10Mbps, 편도 지연(One-way delay) 25ms 환경에서 기본 혼잡 제어 알고리즘인 TCP Cubic 트래픽을 발생
    - 버퍼 튜닝 방식 : 버퍼의 크기를 조절해주는 기능. 기본 방식은 cwnd × 2 로 자동 설정로 계속 늘림
    - 큐잉 방식 : FIFO. 데이터가 나가는 순서를 지정
    - Cubic : 혼잡 제어 알고리즘. 패킷 손실이 날 때까지 속도를 올림.
- 측정 방식
    - 애플리케이션 계층: `write()`, `read()`. 함수 호출 지점
    - 전송 계층 : `tcp_transmit_skb()`, `tcp_v4_do_rcv()`. 데이터를 네트워크로 보내기 좋게 TCP 패킷 단위로 자르고 내려보내는 시점.
    - 하위 계층 : 큐잉 및 물리 계층. 큐에 들어온 시점이나 신호를 보내기 직전의 시점 함수들
- 결과 : 네트워크를 완전히 사용하기 위한 BDP가 44 패킷라 작음에도, 지연 시간이 2.5초. 대체로 TCP 내부에서 발생
    - BDP : Bandwidth-delay product. 네트워크에 동시에 들어갈 수 있는 데이터의 최대량.
        - BDP가 크면 네트워크 지연 시간이 길다고 볼 수 있음
    - 속도도 빠르고 BDP도 적으므로 네트워크 상에서 지연 시간은 짧아야 함. 하지만 2.5초로 긺
    - Cubic은 패킷 손실이 발생할 때까지 속도를 늘리는 데, 버퍼 튜닝 알고리즘은 버퍼를 계속 늘려서 버퍼에 패킷이 머무는 시간이 길어짐.

**능동 큐 관리 (AQM)과 기본 큐잉 방식 성능 비교**

![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image.png)

- 환경 : 저속 유선망, 고속 유선망, 무선망(WiFi), 모바일망(LTE)
- AQM을 적용해봤으나 Endhost 시스템 내부의 딜레이 발생.
- 저대역폭에서 ECN을 사용해봤지만 크게 무의미
    - ECN : 라우터가 혼잡해지기 전에 경고를 보냄.

## ELEMENT Design

**Cubic + 버퍼 튜닝 방식의 재검토의 한계**

Bufferbloat의 근본 원인은 혼잡 제어 알고리즘과 버퍼 튜닝의 상호작용이므로, 이 둘을 재검토하는 것이 자연스러운 해결책처럼 보이지만, 한계가 있음

1. 애플리케이션과 피드백 단절 : 애플리케이션과 피드백을 통해 지연을 조절해야 함. 어떤 앱에서는 지연이 중요하고 어떤 앱에서는 처리량이 중요하지만 이를 구분하지 못함
    - 예를 들어, 버퍼를 작게 만들면 실시간 처리 같은 건 좋겠지만 대용량 파일을 다운로드하는 앱은 (지연과 상관 없이 무조건 속도만 빨라야 함)은 처리량이 떨어짐
2. 클라우드 환경에서 권한 부재 : OS 커널을 수정할 권한이 없음 

**Challenge**

1. 사용자 수준 정보로부터 추론 : 패킷이 애플리케이션을 떠나는 시점과 TCP 계층을 떠나는 시점을 사용자 수준에서 직접 측정할 수 없어 추론해야 함
2. 정확도-오버헤드 tradeoff
3. Non-intrusive integration (비침습적 통합) with TCP application : 라이브러리가 사용하기 쉬우며 기존 TCP 애플리케이션에서 수정이 최소화. 

**Architecture**

![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image-1.png)

- TCP stack : 전송 계층-네트워크 계층-데이터 링크 계층으로 이어지는 통신 프로그램들의 묶음.
1. getsockopt()를 통해 TCP_INFO를 가져와 TCP 통계 획득
    - 전송 및 확인 응답(ACK)된 바이트 수, 혼잡 윈도우 크기(cwnd), 슬로우 스타트 임계값(ssthresh), 왕복 지연 시간(RTT) 등 다양한 통계가 포함
    - 루트 권한이 필요 없음.
2. 애플리케이션의 Sequence Processor에서 두 가지 행동 수행
    1. 송수신된 데이터의 양과 시간 기록
    2. 전송 계층에서 TCP 통계를 수집하는 tcp_info Tracker 실행 
3. 두 계층의 정보는 두 계층의 시퀀스 번호를 짝지어주는 Sequence Mapper에 의해 분석
4. Delay Calculator가 짝지어진 정보를 통해 패킷이 애플리케이션 계층에서 TCP 계층으로 송수신되는 시점 사이의 시간 차이를 추정하며 애플리케이션으로 전송되고 출력 파일에 기록
5. 지연 최소화 알고리즘을 통해 패킷 전송 제어
    - 데이터가 너무 많이 쓰이면 sleep() → packet pacing

**Use Case**

- TCP 지연 시간 튜닝 : 버퍼 튜닝과 혼잡 제어 알고리즘을 변경하지 않고 bufferbloat를 완화
- TCP 기반 화상 회의 : 송수신 지연 시간을 모니터링하여 동기화 간극을 식별. 싱크가 너무 안맞기 전에 선제적으로 조절
- VR 및 원격 의료 : 시각,촉각, 청각적 신호마저 실시간 스트리밍.

## ELEMENT

**송신 측 지연 시간 추정**

- 알고리즘
    
    ![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image-2.png)
    
    - $T_{start}$, $T_{cur}$, $P$, $T$ : 시간을 재기 위한 변수. P는 tcp_info를 가져오는 주기 (10ms)
    - $seq$ : 애플리케이션에서 보낸 누적 바이트 수
    - sendInfo : linked list의 노드
        - bytes : 누적 전송량
        - sendTime : TCP로 보낸 시간
        - next : 다음 노드 포인터
    - data sending thread
        - 데이터를 보낼 때마다 새로운 노드를 만들고 누적 전송량과 시간을 작성.
        - 노드 생성 후 맨 끝에 이어 붙임
        - 패킷을 보내고 seq에 누적량을 더함
        - 반복
    - tcp_info tracking thread
        - P 주기마다 sleep 후 반복
        - $B_{est}$ : 추측된 TCP에서의 누적 전송 데이터
        - if back.bytes <= Best : back은 가장 첫번째로 생긴 노드로 오래된 것. 오래된 것부터 확인
            - D ← T - back.sendTime : 오래된 것부터 확인하여 추측된 TCP에서 전송 바이트가 크다면 딜레이를 계산하고 print함.
            - 계산이 끝난건 메모리 삭제
        - 만약 아직 TCP에서 덜 전송됐다면 sleep

애플리케이션 계층에서 전송된 시간 - 패킷이 TCP 하위 계층으로 전송된 시간을 통해 지연 시간 측정. 

- 시간 측정의 어려움
    - 애플리케이션 버퍼와 TCP 버퍼에서의 전송 크기가 다름.
    - tcp_info는 실제 전송된 데이터의 양과 시간이 적혀 있지 않음

**동작 흐름**

1. 애플리케이션에서 데이터 전송을 할 때, 실제 전송된 누적 바이트 수를 기록하는 seq 변수와 경과 시간 T를 정의.
2. tcp_info를 통해 TCP 계층을 빠져나간 누적 바이트 수를 추정. 
    - tcpi_bytes_acked(상대방이 수신 확인된 바이트수) + (tcpi_unacked(수신 확인되지 않은 조각 수) x tcpi_snd_mss (최대 조각 크기))
        - 상대가 받은 바이트 수 + 네트워크 상에 있는 패킷 x 패킷 최대 크기 (MSS)
3. 짝지어진 바이트에 대해 두 계층의 타임스탬프 차이를 구하여 지연 시간 측정
    - 바이트 수가 정확히 일치하지 않으면 가장 가까운 쌍을 찾음. getsockopt() 함수의 오버헤드 때문에 데이터를 보낼 때마다 TCP 통계를 수집할 수 없음
    - TCP 계층에서 전송된 살짝 과대평가되지만 정확함
- 위를 구현하기 위해 데이터 전송 스레드와 tcp_info 추적 스레드를 도입.
    
    ![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image-3.png)
    
    1. 데이터 전송 스레드 : 데이터를 쓸 때 총 전송 바이트와 전송 시간을 linked list에 기록
    2. tcp_info 추적 스레드 : TCP 계층에서 전송된 바이트 수를 추정. 
        - linked list를 뒤에서부터 탐색
        - TCP 계층에서 추정된 바이트 수를 초과하지 않는 총 전송 바이트 기록을 찾으면 버퍼 지연 시간을 계산하고 ELEMENT로 돌려 보냄
        - 해당 바이트보다 적은 기록들은 리스트에서 삭제

**수신 측 지연 시간 추정** 

- 알고리즘
    
    ![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image-4.png)
    
    - $T_{start}$, $T_{cur}$, $P$, $T$ : 시간을 재기 위한 변수. P는 tcp_info를 가져오는 주기 (10ms)
    - $seq$ : 애플리케이션에서 보낸 누적 바이트 수
    - recvInfo : linked list의 노드
        - bytes : 누적 전송량
        - sendTime : TCP로 보낸 시간
        - next : 다음 노드 포인터
    - tcp_info tracking thread
        - P 주기마다 sleep 후 반복
        - $B_{est}$ : 추측된 TCP에서의 누적 전송 데이터. (받은 세그먼트 수 x 세그먼트 최대 크기)
        - if Best > Bprev : 이전보다 많이 바이트가 도착하면 새로운 노드를 만듦
        - P만큼 sleep
    - data receiving thread
        - 데이터가 올 때마다 실행
        - seq에 데이터의 크기를 더함
        - if back.bytes <= seq : 뒤에서 부터(오래된 것) 확인. 애플리케이션의 누적량 크기가 더 큰 경우 버려짐.
        - 누적량의 크기보다 더 큰 경우는 딜레이 계산하고 print. break를 해서 노드를 버리지 않고 남김.
- 동작 흐름
    1. sk_buff을 통해 TCP 계층의 버퍼를 확인하여 timestamp와 sequence number(누적 수신 바이트 수)를 확인하여 linked list에 기록
        - (수신 세그먼트 수 x 수신자 최대 세그먼트 크기)를 통해 누적 바이트 수를 추측
    2. 애플리케이션에서는 seq num과 애플리케이션의 데이터 기록을 짝지음
        - seq 변수를 통해 애플리케이션의 누적 바이트 수를 정의
    3. 짝지은 바이트 데이터에 대해 timestamp 차이로 지연 시간 측정
        - 정확히 바이트 수가 일치하지 않으면 가장 근사한 쌍을 찾아서 계산

**정확도 (실제 지연 시간과의 차이)**

Linux 커널 프로파일러인 perf, 애플리케이션 소켓 쓰기/읽기에 trace point를 추가. TCP 계층에는 tcp_transmit_skb, tcp_v4_do_rcv 함수를 추적. 4개의 TCP 기반 지연 시간 측정 도구와 비교. 

- tcp_transmit_skb : TCP 버퍼에서 내보내는 함수
- tcp_v4_do_rcv : TCP 버퍼에서 수신하는 함수

**종단 간 지연 시간 최소화**

1. ELEMENT가 제공하는 정보를 바탕으로 애플리케이션의 데이터 전송 속도를 제어. 애플리케이션 버퍼에서 네트워크 대역폭 변동에 따라 불필요한 데이터 삭제 
2. 코드를 변경할 수 없는 기존 TCP 애플리케이션에 대해서는 TCP 연결에서 지연 시간을 최소화하는 알고리즘을 설계 
    - 알고리즘
        
        ![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image-9.png)
        
    - 버퍼 지연 시간 변화에 따라 TCP 송신 버퍼 내 패킷 수를 제어. 지연 시간이 증가하면 버퍼에 머무를 수 있는 패킷 수를 sleep를 통해 감소시킴.
        - 루트 권한 없이 사용 가능
    - $S_{target}$ : 버퍼에 허용되는 최대 패킷 수
        - congestion window를 초과하지 않게 제한.
    - 주기적으로 버퍼 지연을 확인하고 사전에 결정된 지연 임계값($D_{thr}$) 대비 현재 지연 시간의 비율에 따라 $S_{target}$ 을 설정하는 스레드를 만듦
        - 버퍼에 쌓인 데이터 양이  $S_{target}$보다 클 때 sleep
        - write 호출이 너무 제한되면 안되기 때문에 sleep 횟수에도 제한이 있으며, 횟수가 증가함에 따라 sleep 시간을 늘림.
        
        $$
        S_{target} = \min \left( \beta \cdot cwnd, \left( \frac{D_{thr}}{D_{avg}} \right)^{\Delta} S_{target} \right)
        $$
        
        - $\beta \cdot cwnd$ : 최대 한도. 아무리 커도 cwnd의 B 곱보다는 작음
            - cwnd : congestion window 크기
        - $D_{avg}$ : 현재 평균 지연 값
            - 현재 딜레이가 목표치보다 크면 $S_{target}$이 줄어듦
        - $\Delta$ (델타 지수) : 0.25로 설정하여 값이 한 번에 변하지 않고 부드럽게 증가하거나 감소

- 고정 버퍼의 한계
    
    ![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image-10.png)
    
    - 버퍼의 크기를 늘리면 처리량이 증가하지만 지연 시간도 함께 증가함. ELEMENT를 사용하면 동시에 달성.
    
    ![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image-11.png)
    
    - 고정 버퍼에서 사용하는 CUBIC은 패킷 드랍이 발생할 때까지 버퍼를 늘려 처리량은 높지만 bufferbloat 발생
    - ELEMENT와 함께 사용하면 버퍼가 고갈되지 않을 정도로 버퍼 크기를 유지하여 지연 시간을 최소화하면서 처리량을 감소시키지 않음

**ELEMENT API**

![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-06-24%20I%20Sent%20it/image-12.png)

- 신규 애플리케이션 제어 : socket wrapper API르르 사용하여 애플리케이션의 지연 시간을 최적화
    - 호환성 : 똑같은 arguments를 사용
    - TCP 계층의 처리량, 지연 시간, cwnd 크기 등의 상세 정보를 반환
- 기존 애플리케이션 제어 : 코드를 건들일 수 없는 경우.
    - 동적 라이브러리 : TCP 애플리케이션을 실행 하기 전에 LD_PRELOAD 후킹을 사용하여 오버라이드

## Discussion

- **숨겨진 지연 정보의 가시화:** 기존에는 TCP 계층에서 딜레이가 발생해도 위층 앱은 그 사실을 전혀 알 수 없었음. ELEMENT는 이 정보를 앱에 노출하여, 앱 스스로 화질을 낮추거나 전송량을 조절.
- **실시간 TCP 통신의 재발견:** 실시간 통신(VR, 화상회의)에는 보통 UDP를 쓰지만, ELEMENT를 통해 딜레이를 잡은 TCP를 사용하면 방화벽/NAT 통과가 쉽고 남의 트래픽을 방해하지 않는(Fairness) TCP의 장점을 활용
- **매우 낮은 시스템 부하 (Overhead):** 초당 엄청난 패킷이 쏟아지는 극한의 테스트(40개 플로우 동시 실행)에서도 CPU 사용량 증가는 고작 약 4%에 불과하여 실 서비스 적용에 무리가 없습니다.
- **eBPF를 통한 무한한 확장성:** 현재는 TCP 소켓 위주로 추적하지만, 리눅스의 강력한 커널 추적 기술인 eBPF를 도입하면 IP, MAC, 물리 계층은 물론 가상 머신(Hypervisor) 내부의 지연 파악 가능
- **맞춤형 QoS(서비스 품질) 지원:** 향후에는 애플리케이션이 "나는 지연 시간 50ms, 지터(Jitter) 10ms 이하를 반드시 보장해 줘!"라고 ELEMENT에 직접 요구 사항을 전달하는 형태로 발전
    - 지터 : 지연 시간의 변동성
