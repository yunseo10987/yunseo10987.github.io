## 🦥 본문

## Intro.

슬로우 DDoS 공격은 일반적으로 HTTP, FTP, IMAP, SMTP와 같은 애플리케이션 계층 프로토콜을 악용하는 애플리케이션 계층 공격. 애플리케이션 수준에서 서비스 대기열의 대부분 또는 전부를 점유하여, 새로 들어오는 요청들이 폐기

슬로우 HTTP DDoS 공격자는 3-way 핸드셰이크 프로토콜을 사용하여 웹 서버와 연결을 수립한 후, 아주 적은 양의 데이터를 사용하여 연결을 유지 

## Type of Slow Attacks

**Slow HTTP DDoS**

슬로우 HTTP DDoS 공격을 시작하기 위해, 먼저 피해자와 일반적인 TCP 또는 UDP 연결을 수립한 후, 공격자는 웹 서버로 또는 웹 서버로부터 아주 적은 양의 바이트를 보내거나 읽음으로써 수립된 모든 연결을 유지하려고 시도. 

슬로우 HTTP DDoS에는 세 가지 유형이 있습니다: 슬로우 리드(slow read), 슬로우 포스트(slow POST), 그리고 슬로우 겟(slow GET)

공격 근원지를 숨기기 위해 공격자는 연결을 수립하고 유지하는 전송 프로토콜로 HTTPS를 활용

- Slow Read
    
    ![image.png](attachment:79e1d715-cd0b-4b6c-a874-1bba6a53be5b:image.png)
    
    웹 서버에 데이터 자원을 요청한 후, 피해 서버가 아주 낮은 속도로 응답을 보내도록 강제함으로써 정당한 클라이언트가 웹 서비스를 이용하지 못하게 만드는 것을 목표, 웹 서버와 연결을 수립한 후, 공격자는 매우 작은 크기의 TCP 윈도우를 알리면서 자원을 요청
    
    공격자는 알리는 TCP 윈도우 크기를 웹 서버의 버퍼 크기보다 작게 유지하여 지연을 발생시키고, 결과적으로 웹 서버의 버퍼를 발송 대기 중인 응답들로 가득 채우게 합니다.
    
    0바이트의 TCP 윈도우 크기를 알려 웹 서버가 클라이언트의 응답 수신 가능 상태를 무기한 기다리게 만들기도 하지만, 웹 서버에 구현된 타임아웃 메커니즘과 제로 바이트 윈도우 탐지 메커니즘으로 인해 이러한 방식은 탐지
    
    따라서 공격자들은 연결을 유지하고 탐지 메커니즘을 회피할 수 있을 만큼은 크지만, DDoS 상황을 유발할 수 있을 만큼은 작은 다양한 바이트 양을 사용하는 방식을 취할 수 있습니다
    
- Slow POST
    
    ![image.png](attachment:6ec5e1c1-781f-4f47-bb14-93e1e0d757aa:image.png)
    
    슬로우 포스트(slow POST) 공격은 수립된 연결을 장기간 유지하는 속도로 웹 서버에 데이터를 전송
    R-U-Dead-Yet (RUDY) 또는 슬로우 바디(slow body) 공격으로도 알려진 슬로우 POST 공격은 큰 콘텐츠 헤더(content-header) 값을 알리는 HTTP POST 요청을 보내는 것에 의존
    요청을 받으면, 대상 서버는 클라이언트에 의해 연결이 완료되거나 종료될 때까지 데이터 전송을 완료하는 데 필요한 자원을 할당
    
    연결이 활성화되어 있는 동안 지정된 길이의 데이터를 받을 때까지 웹 서버가 기다리기 때문에, 공격자는 비활성 연결의 타임아웃(timeout) 값보다 짧은 간격으로(정기적 또는 무작위로) 소량의 데이터를 서버에 보내는 방식을 사용합니다.
    
    연결이 활성화되어 있는 동안 지정된 길이의 데이터를 받을 때까지 웹 서버가 기다리기 때문에, 공격자는 비활성 연결의 타임아웃(timeout) 값보다 짧은 간격으로(정기적 또는 무작위로) 소량의 데이터를 서버에 보내는 방식
    
    공격이 성공하기 위해서 공격자는 웹 서버에 대해 여러 개의 유사한 연결을 생성하고 동일한 데이터 전송 방법을 시작.
    
    웹 서버의 비활성 연결 타임아웃 값이 30초인 경우, 공격자는 POST 요청에 대해 5메가바이트(MB)의 콘텐츠 길이를 예고해 놓고 15~25초 간격으로 약 20~30킬로바이트(KB)의 데이터
    
- Slow GET
    
    ![image.png](attachment:00868640-7ec6-4b60-ae90-5ae9241a780c:image.png)
    
    슬로우 GET 공격은 슬로우 헤더(slow header) 또는 슬로로리스(slowloris) 공격으로도 알려져 있습니다. 
    
    연결을 수립한 후 웹 서버에 정상적인 GET 요청을 보내지만, 서버로부터 헤더를 받을 준비가 되었다는 신호인 200 OK 메시지를 받으면 공격자는 헤더를 여러 조각으로 나누어 낮은 속도로 전송. 
    
    정상적인 상황에서 헤더는 두 개의 CRLF(Carriage-Return Line-Feed) 문자(“\r\n\r\n”)로 구성되며, 이는 웹 서버에 헤더의 끝과 본문의 시작을 알려 서버가 요청 처리를 시작. 
    
    헤더 요청에서 단일 CRLF 문자는 한 줄의 끝과 다른 줄의 시작을 의미. 
    
    공격 시나리오에서는 두 개의 CRLF 문자가 모두 전송되지 않으며, 이로 인해 피해 웹 서버는 헤더 요청이 완료되기를 무기한 기다리며 연결을 계속 열어두게 됩니다.
    서버의 무기한 대기는 웹 서버의 연결 제한 수치에 도달하게 만들어, 정당한 클라이언트가 보내는 연결 요청이 거부되는 결과를 초래
    
- Slow TCAM
    
    ![image.png](attachment:d488b0df-0270-4e75-8f6c-e20ec9af7afd:image.png)
    
    스위치는 새로운 플로우(flow)가 도착할 때마다 컨트롤러로부터 받은 모든 플로우 규칙을 저장하는 TCAM(Ternary Content Addressable Memory)을 유지. 하지만 TCAM은 1,500개에서 3,000개의 플로우 규칙 항목을 저장할 수 있을 정도로 그 용량이 제한적
    
    슬로우 TCAM 공격은 스위치에 새로운 플로우를 보내 스위치가 플로우 규칙을 요청하게 만들고, 컨트롤러가 스위치에 해당 규칙을 설치하도록 유도
    
    TCAM의 비활성 플로우 규칙 타임아웃 값보다 짧은 간격으로 소량의 데이터를 보냄으로써 해당 플로우 규칙들을 계속 유지 
    
    스위치에 수많은 플로우 규칙을 수립하여 TCAM을 최대 허용 용량까지 채우려 하며, 기존 규칙들이 여전히 활성화된 상태이기 때문에 정당한 트래픽의 새로운 플로우들이 폐기 
    
    낮은 속도로 새로운 플로우를 보내는 대량의 봇(bot)을 모집함으로써 효과적.

### ML

지도 학습(supervised learning) 및 비지도 학습(unsupervised learning) 머신러닝 기술. 

- [13]에서는 slow POST 및 slowloris의 DoS 공격을 탐지하기 위해 5-NN, 나이브 베이즈(Naïve Bayes), 다층 퍼셉트론(MLP), 서포트 벡터 머신(SVM), JRIP, 랜덤 포레스트(Random forest), C4.5 의사결정 나무, 로지스틱 회귀의 사용을 평가 → RF가 좋음
    
    높은 AUC(Area Under Curve)를 달성했는데, 이는 부분적으로 넷플로우(Netflow) 특징 세트를 사용한 덕분
    
    slow POST 및 slowloris 공격에 대해 우수한 탐지 성능. 단일 근원지에서 발생하는 DoS 공격을 사용했다는 점
    
- [17]에서는 슬로우 HTTP 공격을 탐지하기 위해 랜덤 포레스트, KNN, 로지스틱 회귀, SVM, 의사결정 나무, 심층 신경망(DNN) 등 6가지 분류기를 사용
    
    KNN과 의사결정 나무가 높은 탐지율을 기록
    
    불균형 데이터셋을 사용했을 때 KNN의 탐지 시간은 61.21초였으며, 이는 KNN을 사용할 때 슬로우 공격의 즉각적인 탐지가 항상 보장되지는 않음을 의미.
    
- [8] : 전체 패킷 캡처(Full Packet Captures)를 사용하는 대신, [8]에서는 SDN 네트워크에서의 슬로우 리드(slow read) 공격을 탐지하기 위해 8가지 분류기와 함께 넷플로우(Netflow) 특징을 사용. 넷플로우의 사용은 전체 패킷 캡처(FPC)와 관련된 높은 패킷 처리 오버헤드에 비해 오버헤드가 낮다는 점에 기인
    
    랜덤 포레스트 분류기는 슬로우 리드 공격을 탐지하는 최상의 분류기임이 증명. 
    
- [18] : 머신러닝을 통해 이상 징후를 탐지함으로써 slowloris 및 slow POST 공격을 찾아내는 연구가 [18]에서 수행. 사용된 머신러닝 기술은 단일 연결 클러스터링(single linkage clustering), k-평균(k-means), 퍼지 c-평균(fuzzy c-means), 자기 조직화 지도(SOM), 그리고 DBSCAN. 다른 비지도 학습 알고리즘인 k-평균은 네트워크 수준에서 RUDY 공격 탐지를 위해 가장 중요한 특징을 추출할 목적으로 10가지의 서로 다른 특징 순위 산정 방법이 포함된 앙상블 특징 선택 방식을 사용하여 높은 성과.
    
    더 적은 수의 특징을 사용하면 탐지 시간과 분석 정확도가 향상되는 것이 관찰, 선택된 특징들에는 트래픽 크기, 패킷 유사성, 트래픽 속도라는 세 가지 주요 특성을 나타내는 요소. 
    
    5-NN은 7개의 특징이 선택되었을 때 가장 높은 AUC와 가장 낮은 오탐률 값을 가짐으로써 우수한 탐지 성능을 보여줌. 클러스터링 알고리즘이 더 많은 양의 정상 트래픽을 악성으로 분류한다는 점을 고려할 때 그에 따른 오탐률의 증가는 정당화되기 어렵습니다.
    
- [12] : 클라우드 환경에서 슬로우 리드(slow read) 공격을 탐지하기 위해 랜덤 포레스트(random forest) 알고리즘을 사용.
    
    나무의 사전 가지치기(Pre-pruning)를 수행하면 미탐률(false negative rate)이 사전 가지치기를 사용하지 않았을 때의 1.90%에 비해 50.10%까지 증가하는 것으로 나타났습니다
    
    정확도의 경우, 사전 가지치기를 사용하지 않았을 때 99.37%였던 것에 반해 사전 가지치기를 사용했을 때는 83.34%를 기록
    
    하지만 두 경우 모두 오탐률(false positive rate)은 0%로 관찰
    
    랜덤 포레스트에서 나무의 사전 가지치기를 사용하는 것은 나무의 성장을 통해 솔루션이 적절하게 개발되지 못하게 만듭니다
    
    해당 연구에서는 성능 향상을 위해 나무의 수를 늘리는 것이, 해결책을 찾는 데 드는 계산 비용이 탐지율에 악영향을 미치는 시점에 도달하면 정당화되지 않을 수 있다고 언급
    
- [20] : HTTP 개수와 델타 시간(delta time)을 다른 특징들과 함께 사용하여 나이브 베이즈, 나이브 베이즈 다항식, 다층 퍼셉트론, 랜덤 포레스트, 로지스틱 회귀, 그리고 방사 기정 함수 네트워크(RBFN) 머신러닝 분류기로 슬로우 HTTP 공격을 탐지. 나이브 베이즈 다항식 모델이 다른 머신러닝 기술들에 비해 가장 우수한 93.67%의 정확도, 91.49%의 진양성률(true positive), 3.10%의 오탐률을 기록

### Time Series

- [21] : 비매개변수 CUSUM 알고리즘에 기반하여 slow POST, 헤더 및 리드 DoS 공격을 탐지하는 기법이 [21]에서 소개. 이 알고리즘은 관찰된 값들의 분포 변화를 탐지합니다.
2500의 임계치에서 오탐 없이 100%의 탐지율.
선택적 플로우 샘플링(Selective flow sampling)은 샘플링 비율이 20%보다 클 때 가장 높은 탐지율을 기록. 선택적 플로우 샘플링을 통해 얻은 결과는 분석을 위해 큰 플로우가 아닌 작은 플로우를 선택한 덕분.
- [22] : 스펙트럼 분석은 시계열 빈도에 따른 전력 분포에 초점. 신호를 주파수 영역으로 변환하기 위해 이산 푸리에 변환(Discrete Fourier Transform)이 사용. 진행 중인 공격보다 공격의 시작 시점이 더 탐지하기 쉽다는 것이 관찰. 대기 시간 또한 탐지 가능성에 영향을 미치기 때문에, 봇의 대기 시간에 따라 탐지 가능성이 달라지는 것이 확인. 공격자가 고정된 대기 시간을 사용하거나 공격을 시작할 때 연결 요청으로 서버를 폭주시키는 경우, 스펙트럼 분석을 통한 탐지가 가능하다는 것이 관찰
- [23] : 시계열을 랜덤 성분과 추세 성분으로 분리하는 시계열 분해 기법을 사용하였으며, 각각에 누적 합계(CUSUM) 기술과 이중 자기상관 기술을 적용. 32초의 탐지 지연 시간이 기록되었으며, 오탐률(FPR)과 미탐률(FNR)은 각각 4.3%와 9.8%

시계열을 이용한 슬로우 DDoS 공격 탐지 방법은 좋은 탐지율을 달성했으나, 머신러닝 기술과 비교했을 때 여러 요인이 탐지율에 부정적인 영향

### Probabilistic with Distance-based Similarity Metric

- [24] : 유클리드 거리(Euclidean distance) 유사도 메트릭이 [24]에서 슬로우 공격을 탐지하기 위해 채택. 유사도를 계산하기 위해 로그 파일 분석이 사용
- [25] : 유사도 메트릭인 헬린저 거리(Hellinger distance)는 생성된 정상 트래픽과 공격 트래픽의 확률 분포 사이의 거리를 측정. 공격자가 벤치마크로 사용되는 정상 트래픽과 유사한 확률 분포를 가진 패킷을 생성할 수 있다면 탐지 시스템의 우회는 불가피
- [26] : 카이제곱(Chi-square) 통계 또한 낮은 속도의 DoS 공격을 탐지하는 데 사용. 적절한 임계값과 간격 시간을 선택하는 것은 어려움이 따랐는데, 간격 시간이 늘어나면 재현율(recall rate)은 향상되지만 높은 오탐률을 유발하고, 반대로 간격 시간을 줄이면 재현율은 떨어지고 오탐률은 개선

확률 및 거리 기반 유사도 메트릭의 사용은 슬로우 DDoS 공격을 탐지하는 데 있어 아직 효과적임이 입증되지 않았습니다.

### Performance Model

- [11] : 패킷 간 도착 시간(inter-arrival time)과 윈도우 크기 분석이 슬로우 HTTP DDoS 공격을 탐지하기 위해 [11]에서 사용. 정상 트래픽, slowloris, RUDY, 그리고 슬로우 리드 공격의 클라이언트-서버 간 통신 평균 윈도우 크기는 각각 34041, 14123, 14034, 7241로 확인되었으며, 서버-클라이언트 간 통신에서는 각각 27022, 6854, 6856, 0으로 기록. slowloris와 RUDY 공격의 평균값이 서로 가깝게 나타나는 것을 관찰할 수 있는데, 이는 두 공격의 유사성.
    
    클라이언트-서버 간 평균 패킷 델타 시간은 정상, slowloris, RUDY, 슬로우 리드 공격에서 각각 302.28, 75.16, 74.123, 339.28 ms였으며, 서버-클라이언트 간의 시간은 151.12, 0.115, 0.561, 28.759 ms로 기록
    
- [15] : OpenStack 클라우드 플랫폼에서의 슬로우 HTTP DDoS 공격에 대한 해결책. 패킷 사전 모니터링 모듈이 패킷의 동작을 식별하고 이를 분류기 영역(classifier zone) 모듈로 전달. 모든 클라이언트는 특정 조건을 위반하기 전까지 허용 목록에 배치. 평균 네트워크 지연 시간은 클라이언트에 5번의 핑(ping)을 보내 계산되며, 평균 응답 시간은 클라이언트가 핑 메시지에 응답하는 시간을 고려하여 계산. HTTP 요청 간의 지연 시간이 계산된 네트워크 지연 시간의 5배를 초과하면 해당 클라이언트는 차단 목록 영역. 5배 기준은 애플리케이션의 처리 시간을 고려.
    
    0바이트의 TCP 윈도우를 빈번하게 알리는 행위는 모니터링되어 차단 목록에 배치
    
    타임아웃 값의 80%가 경과했을 때 웹 서버로 전송되는 POST 또는 GET 요청은 공격자로 간주되어 차단 목록에 배치
    
- [27] : SDN 네트워크에서 슬로우 공격 탐지를 돕는 연결 임계값이 [27]에서 조사. 웹 서버의 개방된 연결 수가 처리 중인 동시 연결의 예정된 임계값을 초과할 때 불완전한 HTTP 요청이 발생하면 슬로우 공격으로 탐지.
- [16] : TCAM이 가득 차서 새로운 규칙을 설치할 수 없을 때 SDN에서 발생하는 TABLE FULL 메시지는 [16]에서 TCAM 공격을 탐지하는 데 활용되었으며, 이는 결과적으로 완화 메커니즘을 활성화.
- [28] : 역방향 프록시(Reverse proxy)가 [28]에서 slowloris 공격을 완화하고 서버의 스트레스를 측정하여 공격을 탐지하는 데 사용. 역방향 프록시는 요청이 완료될 때까지 원래 서버를 대신하여 요청을 처리.
- [29] : 중앙 처리 장치(CPU) 사용률 및 시간, 작업량, 디스크 사용률 및 시간, 대기 시간 및 처리량을 사용하여 공격을 나타내는 기준선(baseline)을 형성하는 성능 모델의 수립이 [29]에 의해 탐구. 기준선을 언제 수립해야 하는지에 대한 딜레마가 그들의 모델에서 식별된 장애물. 아마도 생성된 기준선이 공격이 진행 중일 때 만들어졌을 수 있으며, 이로 인해 수립된 기준선을 사용하여 이후의 공격을 쉽게 탐지하는 것이 어려워질 수 있습니다. 성능 기반의 슬로우 DDoS 공격 탐지 모델은 공격 탐지에 유용한 것으로 입증되었으나, [29]에서 명백하듯이 문제가 없는 것은 아닙니다.공격과 정상 트래픽의 동적인 특성을 고려할 때 적절한 임계값을 선택하는 것은 수행하기 어려웠습니다.

## Discussion

머신러닝 탐지 기술은 컴퓨터 네트워크에서 슬로우 DDoS 공격을 탐지하는 데 효과적이고 효율적임이 입증. 랜덤 포레스트의 계산 오버헤드와 불균형 데이터셋이 존재할 때 KNN의 느린 탐지 시간은 이들의 단점. 

성능 모델 및 시계열 기술을 이용한 슬로우 공격 탐지는 달성된 결과 측면에서 표 I에 나타난 바와 같이 머신러닝 기술에 뒤처져 있으며, DDoS 공격 탐지 방식에 대한 개선이 필요. 

장점은 특징을 쉽게 추출하고, 더 빠르게 탐지를 수행하며, 공격 탐지를 위해 외부 모듈에 의존하지 않기 때문에 탐지 시스템의 자원을 절약할 수 있다는 점.

거리 기반 유사도 기술과 함께 확률을 사용하는 방식은 아직 눈에 띄는 결과를 내지 못했습니다.

## Conclusion

슬로우 공격 탐지에 대한 적절한 연구의 부족은 CAIDA, NSL-KDD, DARPA 데이터셋이 존재하는 볼류메트릭 DDoS와 달리, 슬로우 공격에 대한 표준 데이터셋이 부재하다는 점에서도 명확히 드러납니다. 

데이터셋이 없는 상황에서 슬로우 DDoS 공격 연구자들은 slowHTTPTest, [slowloris.py](http://slowloris.py/), OWASP switchblade 등과 같은 도구를 사용하여 공격을 시뮬레이션함으로써 자체적인 데이터셋을 만드는 데 의존

특징 선택과 더불어 머신러닝 기술의 파라미터 조정에 대한 탐구도 이루어져야 합니다. 테이블 오버플로우 공격이 피해를 입힌 후에 탐지하고 완화하는 것은 효율적이거나 예방적이지 않으므로, 슬로우 테이블 오버플로우 공격 탐지 및 완화에 관한 연구가 필요
