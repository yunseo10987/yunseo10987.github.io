## 🦥 본문

# Facet

비디오를 화상 회의 플랫폼을 통해 스트리밍하여 검열을 우회하는 시스템

## Motivation

국가 기관에 의해 검열자들이 정보 접근을 차단하는 경우가 많아짐. 

**기존 우회 도구들의 한계**

- Tor 및 기타 프록시
    - DPI(심층 패킷 검사)를 통한 정교한 프로토콜 핑거프린팅
    - active probing attack를 통해 의심되는 릴레이가 Tor 프로토콜에 참여를 하는지 확인
- 프록시 스테가노그래피 : 프록시 연결을 무해한 커버 프로토콜처럼 보이게 만드는 방법
    - 예를 들어, 프록시 연결을 VoIP 통화처럼 보이게 하려 시도하거나 콘텐츠 공유 사이트에 게시된 사진 속에 정보를 숨김
    - **불일치**
        - 에뮬레이션 불일치 : 클라이언트나 프록시가 커버 프로토콜을 완벽하게 구현하지 못함.
            - HTTP 서버와 동일한 방식으로 요청에 응답하지 않거나 TCP control 연결을 시뮬레이션 하지 않음
        - 채널 불일치 : 커버 프로토콜이 channel behavior에 프록시 프로토콜과 다르게 응답함.
            - channel behavior : 통신에서 채널 또는 프로토콜이 환경 변화나 간섭에 반응하여 작동
            - 예를 들어, 검열자가 패킷을 드롭할 때, VoIP 프로토콜은 내성이 있지만 프록시는 패킷 손실에 내성이 없음.
        - 콘텐츠 불일치 : 커버 프로토콜이 프록시 프로토콜과 데이터의 내용이나 트래픽 특성에 있어서 차이가 나서 발생
            - 예를 들어, VoIP 채널을 통해 음성 신호 대신 다른 데이터를 보내는 데, 트래픽 분석에 의해 식별될 수 있음.

**가정**

검열자가 부수적인 피해를 피하기 위해 커버 프로토콜(스카이프)의 모든 또는 대부분의 세션을 무차별적으로 차단하기를 꺼림. TLS 및 IPsec과 같이 널리 사용되는 암호화 프로토콜을 허용

**기능** 

- 에뮬레이션 일관성 : 비디오가 실제 양방향 스카이프 통화를 통해 전송되기 때문에, 식별을 허용하는 구현 간에 차이가 없음
- 채널 일관성 : 비디오용으로 의도된 채널을 통해 비디오를 전송하므로, Facet 세션에 대한 모든 중단은 일반 통화에 대한 동일한 중단을 야기
- 콘텐츠 일관성 : 임의의 비디오는 화상 회의 통화와 다른 특성을 가질 수 있으며, 이는 패킷 크기에서 탐지 가능한 차이를 유발.
    - Wright 등[31]의 접근 방식과 유사한 이진 분류기를 구현했으며, Skype를 통해 전송된 수정되지 않은 YouTube 비디오가 Skype 통화와 구별될 수 있음을 보여줌 →  Facet 서버가 요청된 비디오를 무작위로 선택된 화상 회의 통화 내에 프레임하는 "비디오 모핑(video morphing)"을 도입. Facet 통화의 90%를 인식할 수 있는 분류기의 오탐율(false positive rate)을 거의 40%로 증가시킴
- 관찰 불가능성 : 화상 회의 스트림은 전송 중에 암호화되므로, 검열자는 Facet 세션을 탐지하기 어렵. 화상 회의 소프트웨어(또는 통화가 피어 간에 직접 라우팅되지 않는 경우 서버)가 손상되더라도, 무작위 비디오 모핑의 사용은 검열자가 모든 비디오 통화를 실시간으로 디코딩하고 분석하여 Facet 세션을 탐지하도록 강제
- Facet의 처리량은 실시간 비디오 전송을 제공하기에 충분히 높습니다.
    - 대부분의 은닉형 반검열 도구는 일반 웹 브라우징을 위해 설계되었으며, 클라이언트를 위한 대역폭이 제한적인 경우가 많음
    - Facet은 클라이언트를 위한 실시간 비디오 서비스 제공을 목표로 하며, 화상 회의 서비스와 동일한 처리량을 달성
- 제공자 독립적 : Facet의 에뮬레이터 장치는 화상 회의 시스템과 독립적으로 구축되므로, Facet은 Google Hangout, Skype, FaceTime 또는 QQ와 같은 모든 화상 회의 플랫폼에서 널리 채택. 다양한 화상 회의 시스템에 접근할 수 있는 사용자에게 접근성을 제공할 뿐만 아니라, 단일 프로토콜 또는 구현을 대상으로 하는 차단을 회피할 수 있는 능력을 제공
- 클라이언트 측 배포가 필요 없음. Facet 클라이언트는 (종종 차단되는) 클라이언트 소프트웨어를 설치하거나 서버와 사전에 비밀을 공유할 필요가 없습니다. 유지 관리하기 쉽게 만듭니다. 소프트웨어 업데이트는 검열 지역 외부의 서버에서만 적용

## Threat Model

- In-depth Traffic Analysis (심층 트래픽 분석) : 의심스러운 트래픽을 감지하고 원치 않는 연결의 징후가 발견되면 이를 차단
    - 키워드 필터링, 정적 IP 주소 필터링, 프로토콜 핑거프린팅 등
    - 패킷 길이 같은 통계적 기법을 사용하여 은닉 트래픽을 분석
- Proactive Detection (사전 탐지) : 검열자가 차단을 위해 우회 도구의 사용자로 행동
    - proactive probing (사전 프로빙) : 검열자는 잠재적인 우회 프록시(우회 프록시로 의심되는 IP)에 프로브(패킷)를 보내며, 우회 서비스를 제공하면 차단
    - Enumeration (열거) : 가능한 IP 주소 범위나 알려진 프록시 목록을 가지고 접속 시도. 우회 서비스를 제공하는 지 확인 후 차단
- Active Interference (능동적 간섭) : 잠재적인 우회 시스템의 세션에 패킷 지연, 삭제, 주입.
    - Skype 세션에서 ACK를 삭제하여 실제 Skype 세션에 거의 간섭 없이 SkypeMorph를 방해

### Design Goal

- Consistency (일관성) : 프로토콜, 아키텍처 및 콘텐츠 측면에서 실제 시스템과 일관
- Unblockable (차단 불가능성) : 트래픽 분석, 서비스 거부 공격 및 능동적 간섭에 대응
- Scalability (확장성) : 특정 화상 시스템과 독립적 → 광범위한 화상 회의 시스템에 적용 가능하게 해야 클라이언트가 쉽게 만들 뿐 아니라, 검열자가 차단하기 어려움
- 실시간 전송

## BACKGROUND

**Videoconferencing Systems (화상 회의 시스템)** 

- UDP
- 아키텍처
    - P2P(peer-to-peer) : 트래픽 클라이언트 간에 전송.
    - 중앙 집중식 : 트래픽을 중앙 서버가 중
- 코덱 : 디지털 데이터 스트림을 인코딩하고 디코딩하는 데 사용
    - 효율을 위해 VBR (가변 비트율)을 사용
    - VBR은 전송되는 패킷 크기 분석으로 콘텐츠에 대한 정보를 누설함

**Parrot Circumvention Systems (앵무새 우회 시스템)** 

- SkypeMorph : Tor 클라이언트와 브리지 간의 연결을 Skype 통화 트래픽으로 위장.
    - 브리지와 클라이언트 간의 Skype 영상 통화를 위장 목적으로 시작한 후, 실제 Skype 연결을 끊고 Skype가 아닌 UDP를 통해 Tor의 TCP 트래픽을 전송.
    - 패킷화 모듈을 통해 Skype UDP 트래픽처럼 보이게 만듦
- StegoTorus : Tor 프로토콜을 난독화.
    - HTTP Request에서 URL 및 쿠키 필드에 데이터를 숨김
    - HTTP Response에서 PDF 및 Flash와 같은 첨부 파일에 데이터를 숨김
- CensorSpoofer : IP Spoofing을 사용하여 차단되지 않는 웹 브라우징 서비스를 제공.
    - 요청 URL은 경량이므로 저용량 채널(ex. 이메일)을 사용하여 전송
    - 응답 트래픽은 대용량이지만 IP spoofing으로 검열자를 속임
    - 프록시의 IP 주소가 요청과 응답에서 노출되지 않음
- FreeWave : Skype 음향 채널을 사용하여 트래픽을 전송. 클라이언트는 웹 요청을 음향 신호로 변조하여 VoIP 서비스에 의해 전달. FreeWave 서버 측에서는 음향 신호를 복조하여 요청을 추출. 모뎀 트래픽으로 전달

**Attacks on Circumvention Systems (우회 시스템에 대한 공격)**

- 모방 결함 : 실제 시스템의 복잡성과 독점적인 특성으로 모방이 어려움
    - Side channel 모방 실패 : 실제 시스템은 일반적으로 트래픽 제어, 사용자 로그인 등을 위한 측면 채널을 가짐.
        - side channel : 시스템의 전력 소비, 실행 시간, 전자파 방출, 혹은 소리 등의 물리적인 특성을 관찰함으로써 민감한 정보를 얻어내는 공격
    - 반응 모방 실패 : 오류 및 네트워크 조건을 처리하는 데 있어 실제 시스템과 다르게 동작.
        - StegoTorus : 검열자가 HTTP 요청을 보낼 때 다른 오류 메시지를 반환
        - CensorSpoofer : 검열자의 프로브에 응답할 때 실제 SIP 클라이언트와 다르게 동작
    - 모방 오류:  SoM 필드 부족으로 Skype UDP 패킷을 잘못 모방.
        - StegoTorus는 xref 테이블이 누락된 잘못된 PDF를 생성.
        - SkypeMorph와 StegoTorus 모두 생성된 추적을 재사용하여 Skype 트래픽과는 다름
- 불가피한 불일치 :  콘텐츠와 채널 측면에서 실제 프로토콜과 프록시 프로토콜 간의 불가피한 불일치
    - 콘텐츠 불일치
        
        FreeWave에서는 사람의 음성 대신 변조된 음향 신호가 VoIP를 통해 전송. VoIP 프로토콜은 일반적으로 VBR 인코딩을 채택하므로, 패킷 길이는 전송된 내용을 드러낼 수 있음. 
        
    - 채널 불일치
        - SkypeMorph와 FreeWave 모두 Tor TCP 패킷 또는 동기화 프레임을 전송하기 위해 안정적인 채널을 필요.
        - VoIP는 UDP를 사용하기 때문에 패킷이 드랍되도 큰 문제가 없음
        - 공격자가 패킷 드롭핑을 하면 VoIP는 문제가 없지만  SkypeMorph와 FreeWave는 동기화 해제가 되거나 무기한 중지가 될 수 있음

## FACET Design

![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-04-25%20Facet/image.png)

클라이언트의 동영상 URL 탐색

1. 암호화된 비디오 검색 : 차단되지 않는 웹 검색 서비스 이용
2. 동영상 구독 : 정기적으로 수신되는 이메일을 통해 구독 중인 영상의 제목과 URL 정보를 확인하고 탐색 
3. 검색 엔진 프록시 : Facet에서 이메일 터널을 사용하는 검색 엔진 프록시
    - 클라이언트가 검색 키워드를 Facet 서버로 이메일을 보냄
    - 서버가 검색 결과를 가져와서 해당 페이지의 스크린샷을 클라이언트에게 이메일로 보냄
    - 이메일 터널의 관측 불가능성 덕분에 보안이 보장되고 Facet 서버의 이메일은 공개적으로 배포

**동작 흐름** 

1. 서비스 발견 : Facet 서버는 서비스 검색을 위해 컨퍼런스 ID를 배포. 
    - 중앙 집중식 : 컨퍼런스 ID는 공개 배포. 검열자에게 ID가 공개돼도 차단되지는 않음
        - 이유 1 : 통신 세션 전체가 암호화돼서, 특정 세션과 컨퍼런싱 ID를 연결하여 차단할 수 없음
            
            트래픽이 암호화되어 있어서, 정상 트래픽과 Facet 트래픽을 구분하지 못함. 어떤 통신이 컨퍼런싱 ID를 사용한 지 모름 
            
        - 이유 2 : 능동적으로 탐색하더라도 트래픽 자체는 화상 회의 중앙 서버를 거치기 때문에 중앙 서버의 IP만 확인할 수 있고 Facet 서버의 IP는 파악하기 힘듦
    - 분산형 화상 회의 시스템 : Facet 서버 ID는 비공개로만 배포.
        - 공개되는 경우 검열자는 서비스를 능동적으로 탐색하여 Facet 서버 IP 주소를 정확히 파악하고 차단
2. 접속 요청 : 클라이언트는 서버에 연락처 추가 요청을 보냄. 서버가 요청을 수락한 후, 초기 연결을 설정. 
3. URL 전달 : 클라이언트는 검열된 비디오의 URL를 인스턴트 메시지 또는 이메일을 통해 서버로 보냄
    - 서비스 운영사가 검열자와 공모하는 경우가 있을 수 있으므로 이메일 전달이 더 안전함
4. 에뮬레이터 실행 : 서버는 클라이언트의 요청을 추출하고, 오디오 및 비디오 에뮬레이터를 시작하여, 클라이언트에게 화상 전화를 함 
5. 비디오 처리 파이프라인 : 서버는 URL을 Facet 파이프라인으로 전달
    - 비디오를 다운로드 → 디코딩 → 크기 조절 후 에뮬레이터 장치로 스트리밍

            - youtube-dl로 주소 추출
            - souphttpsrc를 통해 다운로드 하며 전체 동영상이 다운로드될 때까지 기다리지 않고 비디오 핸들러에 전달
        - 비디오 핸들러
            - decodebin2로 비디오 및 오디오 스트림이 분리되고 에뮬레이터에 배치
            - videoscale, videorate 및 ffmpegcolorspace와 같은 추가 요소를 사용하여 색 공간, 해상도 및 프레임 속도 등을 변경
        - 에뮬레이터
            - 카메라 에뮬레이터 : 가상의 카메라 장치 v4l2loopback 사용
            - 마이크 에뮬레이터 : 가상의 마이크 장치 pactl 사용
6. 영상 시청 : 화상 회의 요청을 수락한 후, 클라이언트는 화상 회의 세션에서 비디오를 시청
7. 재생 제어 : 클라이언트는 비디오 재생 또는 비디오 속도 조정을 위해 서버에 제어 명령을 보낼 수 있음.
8. 세션 종료 : 비디오가 끝나거나 클라이언트가 컨퍼런스를 종료하면, Facet 서버는 에뮬레이터를 파괴하고 세션을 종료

**보안**

- 영상 시청의 경우 데이터 흐름이 단방향적일 수 있기 때문에, 이를 통해 검열자가 차단할 수 있음. 클라이언트가 카메라를 활성화하여 양방향 트래픽을 강제로 생성하여 우회
- DoS 공격 방지를 위해 사용자는 서버의 ID로 연락처 추가를 보내고 서버도 사전에 정해진 규칙에 따라 승인
    - 또한 등록된 클라이언트 ID에 사용량 제한을 적용

## 트래픽 분석

패킷 길이의 통계적 분포 (n-gram)을 분석하여 높은 확률로 구분  

분류기 : VoIP 통화에서 언어를 구별하기 위한 χ2 분류기. 패킷 길이 n-gram을 통해 분류. 

- 트래픽 지연은 안정적이지 않아서 불가
- 패킷 길이는 차원의 저주 때문에 이산화를 시킴 (K)

![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-04-25%20Facet/image-2.png)

Region 1 : 실제 화상 회의는 100~200 구간이 더 많음. 화상 회의 오디오의 품질이 낮고, 오디오 전송에 더 짧은 패킷을 사용. 샘플링 속도의 불일치로 더욱 입증

- 샘플링 속도 차이 (아날로그 신호를 1초당 몇 번 쪼개서 디지털로 저장하는냐) : 유튜브는 44100Hz, 채팅은 11,500Hz로
- 비트폭 (쪼개진 소리 한 조각의 정보를 얼마나 상세하게 표현하느냐) : 채팅은 16비트, 유튜브는 32비트가 두 채널

Region 2 : 화상 회의는 유튜브 비디오보다 400~600바이트 및 800~1000바이트 범위의 패킷을 더 많이 보유

- 채팅 비디오는 저속 모션 특성에 의해 인코더가 데이터를 효율적으로 압축하고 특정 크기의 패킷들이 일정한 패턴이 발생
- 유튜브 비디오는 화면 전환이 빠르고 움직임이 많아 데이터의 양과 채팅 영상과 다름

**모핑**

트래픽 분석에 대응하기 위해 도입. 패킷을 조작하는 것은 검열자에게 탐색 당하기 쉬움

- 패킷 조작 : 패킷의 크기를 맞추기 위해 데이터를 채워 넣거나 전송 타이밍을 인위적으로 늦추는 등의 방식

![image.png](https://yunseo10987.github.io/assets/images/posts_img/2026-04-25%20Facet/image-3.png)

**오디오 모핑** 

유튜브 오디오를 실시간으로 리샘플링하여 화상 채팅의 품질을 흉내냄 

- 유튜브의 2채널을 단일 채널로 변경하고 비트폭을 16비트로 설정
- 샘플링 속도를 3000Hz일 때 효과적임.

**비디오 모핑** 

H.264에서 각 프레임은 매크로블록이라고 하는 여러 개의 작은 정사각형 블록으로 나뉘는데, 검열된 비디오를 전체 화면 중 일부 블록에 유튜브 영상을 넣고, 나머지는 실제 화상 채팅 영상으로 채움 → 실제 화상 통화와 패턴이 비슷

모핑 레벨 (s) : s가 작을 수록 유튜브 영상을 차지하는 비중이 줄어들어 보안성은 높지만 시청 품질은 낮아짐. 

## Security Analysis

**에뮬레이션 불일치를 이용하는 공격** : 화상 회의 시스템을 직접 사용하므로 에뮬레이션 결함이 존재하지 않음

**콘텐츠 불일치를 이용하는 공격** : 콘텐츠 일관성이 있음. 비디오(또는 오디오) 채널을 사용하여 비디오(또는 오디오)를 전송. 또한, 모핑으로 트래픽 분석까지 불가능 

**아키텍처 불일치를 이용하는 공격**

- 중앙 집중식 : Facet는 중앙 서버를 통해 클라이언트와 통신하므로 아키텍처 분석으로 화상 회의와 Facet를 구분할 수 없음
- P2P : 검열 지역 밖에 있는 여러 개의 프록시를 거쳐 클라이언트와 연결하는 터널링 전략을 사용.

**채널 불일치를 이용하는 공격** : 채널 일관성을 가짐. Facet는 패킷 지연,손실, 중복 같은 것들이 실제 화상 회의와 같은 품질 저하를 겪음 

**프라이빗 시청** : 시청 기록이나 정체가 Facet 서버 운영자에게 노출되는 전략 (또는 검열자가 운영하는 가짜 서버)

- 가명 ID 사용 : 별도의 계정으로 서비스를 이용. Facet 서버도 시청 기록을 클라이언트가 아닌 가명에만 연결
- IP Geolocation 방어 : 서버가 사용자의 IP 주소를 추적하여 위치를 파악하는 공격
    - 중앙 집중식의 경우 : 중앙 서버를 거치므로 IP를 알 수 없음
    - P2P : 사용자가 화상 회의를 시작하기 전에 검열된 지역 안팍에 위치한 프록시를 설정

**DoS** : 서버 ID가 공개적으로 배포될 경우 DoS 공격의 위험. CAPTCHA나 각 클라이언트에 대한 사용 제한으로 무력화 

**화상 회의 제공 업체가 검열자와 공모하는 경우**

사용자의 온라인 채팅에 대한 필터링과 감시를 하는 경우가 있을 수 있음

- URL 제출을 이메일로 제공
- 오디오/비디오 필터링은 비싸고 어려움
- 제공 업체와 독립적이므로 전환 가능함

**자주 시청되는 동영상 Fingerprints** 

인기 있는 유튜브 영상은 여러 사용자가 반복해서 시청하여 Fingerprint가 발생할 수 있음. 

- 비디오 모핑 시 사용하는 영상을 매 세션마다 다르게 선택함
- 오디오의 샘플링 속도를 미세하게 조정하여 트래픽의 통계적 특성을 변형시킴
- 영상 뒤에 무작위 영상을 이어 붙여 동영상 길이를 매번 변형.
